1.    Introduction

1.1    Purpose

This document describes the requirements for a software application named Annotation Is Relatable (AIR) that enables users to annotate existing Portable Document Format (PDF) documents with text and boxes. This document enumerates requirements related specifically to annotation creation, modification, deletion, saving, and loading without restating requirements for PDF document contents and structure separately documented in ISO 32000-1:2008 "Document Management — Portable Document Format" Part 1.

1.2    Document Conventions

By convention in this document, higher-level requirements in this document are inherited by detailed requirements. In other words, detailed requirements elaborate higher-level requirements without superseding higher-level requirements. Higher-level requirements are denoted by requirement identifies of the form REQ-NNN000 where N is a digit and the identifier contains three or more zeros in in the least significant numeric positions. Detail requirement identifiers have the form REQ-NNNMMM where N and M are digits, and the detail requirement elaborates the corresponding higher-level requirement, REQ-NNN000. For example, a hypothetical requirement identified as REQ-005010 elaborates a hypothetical requirement identified asREQ-005000.

By convention in this document, words in italics are terms defined in Appendix A Glossary and may have narrower definitions in the context of this document than in general use. For example, the word text has a more constrained definition within this document than in general use.

Requirements that contain the word shall constitute essential or mandatory requirements for any acceptable implementation of AIR. Requirements that contain the words will or may describe intended features or optional features that may or may not be present in an acceptable implementation of AIR. In all cases, requirements containing the word shall have priority and supersede requirements that contain the words will or may. In other words, an acceptable implementation of AIR may sacrifice implementation of will or may requirements as necessary to achieve all shall requirements.

1.3    Intended Audience and Reading Suggestions

This document is intended for use by software developers, instructors, and other stakeholders interested in the features and capabilities of AIR. Readers should refer frequently to Appendix A Glossary to avoid misinterpretations of terms that have specific limited definitions in the context of this document. Pay attention to the product scope described in Section 1.4 Product Scope. Reference ISO 32000-1:2008 "Document Management — Portable Document Format" Part 1 for contextual information about requirements described in this document. Enumerated requirements are in Section 4 System Features, Section 5 Other Nonfunctional Requirements, and Section 6 Other Requirements.

1.4    Product Scope

AIR is a software tool that enables users to annotate existing Portable Document Format (PDF) documents with text and boxes. AIR is intended to be cross-platform and usable with current versions of the Java Runtime Engine (JRE), Mac OS, and Microsoft Windows at the time this document is published. The linked Figure 1 Mockup is a mockup depicting the types of annotations that may be created, modified, deleted, saved and loaded by AIR. Please reference the Glossary in Appendix A for definitions of terms like Modify. The blue boxes and blue text in the lower right corner in Figure 1 Mockup depict annotations. The black text in Figure 1 Mockup depicts the original (non-annotation) content of a PDF document that has been annotated.

2.    Overall Description

2.1    Product Perspective

AIR is software application that interoperates with PDF documents to enable user creation, deletion, modification, saving, and loading of annotations consisting of boxes and text. Intended uses include instructor annotation of student submitted PDF documents to include review findings, grade markings, boxes around non-annotation content and/or annotations, and general text comments.

AIR may make use of existing third party PDF processing support libraries such as Apache PDFBox or ITEXT.

2.2    Product Functions

AIR enables users to annotate existing PDF documents with text and boxes by performing the following operations:

      Load an existing PDF document into AIR

      Insert text annotations

      Insert box annotations

      Modify previously inserted box annotations. Ref: precise definitions of terms like Modify in Appendix A, Glossary

      Modify previously inserted text annotations. Ref: precise definitions of terms like Modify in Appendix A, Glossary

      Delete previously inserted box annotations

      Delete previously inserted text annotations

      Save a previously loaded PDF document including any inserted box or text annotations

2.3    User Classes and Characteristics

AIR is intended for use by anyone who has reason to view a PDF document on any of the systems/platforms supported by AIR. PDF documents annotated using AIR remain valid PDF documents that may be viewed, printed, or otherwise used in the same ways unannotated PDF documents are used.

2.4    Operating Environment

AIR is intended for use on systems/platforms with the current versions of the Java Runtime Engine (JRE), Mac OS, and LInux at the time this document is published. AIR is intended for use in common office environments. AIR may not be used as a component in any safety critical systems such as aircraft, medical devices, or nuclear devices.

2.5    Design and Implementation Constraints

AIR will be developed using Java 8.

AIR will follow Java language coding conventions consistent with any existing third party PDF processing support libraries such as Apache PDFBox or ITEXT. In particular AIR source code will conform to recommendations in Code Conventions for the JavaTM Programming Language from 1999 except when Java language changes or updates since 1999 conflict with the 1999 conventions or where Oracles javac –Xlint provides conflicting guidance.

AIR will be Open Source licensed using either or both of GNU Affero General Public License version 3 or Apache License v2.0.

AIR will be developed by Wright State University graduate students enrolled in CS-7140-C01 Summer 2016.

AIR and associated work products including this document will be submitted for a course grade by the conclusion Wright State University Summer 2016 Full Term.

AIR will operate and will be demonstrated on typical Wright State University owned PCs Running Microsoft Windows or Mac OS.

AIR will be delivered with the following documents:

      Users Manual

      Test Report

2.7    Assumptions and Dependencies

AIR may be implemented to use existing third party PDF processing support libraries such as Apache PDFBox or ITEXT.

4.    System Features

4.1    PDF Annotation

4.1.1 Description and Priority

The sole feature of AIR is PDF annotation which includes loading PDF documents, displaying some or all components of loaded PDF documents to the user, creating annotations, modifying annotations, deleting annotations, and saving PDF documents that include annotations all while preserving non-annotation components in the PDF document.

4.1.2 Stimulus/Response Sequences

A typical use case involves the following sequence of operations:

1.     User will direct AIR to load a PDF documents selected from within a file system by a user.

2.     User will use AIR to view displayed non-annotation components of the loaded PDF document.

3.     User will direct AIR to create one or more annotations consisting of text and/or boxes and use AIR to view the created annotation along with displayed non-annotation components in the same document.

4.     User will use AIR to modify created annotations by changing the character sequences in annotation text or by resizing annotation boxes or my repositioning annotation text or by repositioning annotation boxes. Ref: the precise definition of Modify in the Appendix A, Glodssary.

5.     User will optionally direct AIR to delete previously created annotations.

6.     User will optionally direct AIR to save the annotated PDF document.

7.     User will later direct AIR to load the previously saved annotated PDF document.

8.     User will use AIR to view previously created and saved annotations along with displayed non-annotation components of the loaded PDF document.

9.     User will continue at step 3.

 

4.1.3 Functional Requirements

REQ-041000: AIR shall enable user to load PDF standard conforming documents.

REQ-041010: AIR shall display all text in loaded PDF standard conforming documents.

REQ-041015: AIR shall display all boxes in loaded PDF standard conforming documents

REQ-041020: AIR shall display text in conformance with PDF Text-Positioning Operators as defined in Section 5.3.1 of "PDF Reference sixth edition" version 1.7 November 2006.

REQ-041030: AIR shall enable user to create annotations in loaded PDF standard conforming documents.

REQ-041040: After creating annotations in a loaded PDF standard conforming documents, the resulting modified documents shall remain PDF standard conforming documents.

REQ-041050: AIR shall enable user to save annotated PDF documents.

REQ-041060: AIR shall enable user to load previously saved annotated PDF documents.

REQ-041070: AIR shall enable user to position created annotations at any positions in the PDF User Space Coordinate System as defined by Section 4.2.1 of "PDF Reference sixth edition" version 1.7 November 2006.

REQ-041080: AIR shall enable user to delete any annotations previously created in PDF standard conforming documents.

REQ-041090: After deleting annotations previously created in PDF standard conforming documents, the resulting modified documents shall remain PDF standard conforming documents.

REQ-041100: When annotations are created in pre-existing PDF standard conforming documents, AIR shall preserve all non-annotation components of the pre-existing PDF standard conforming documents whether or not the components are displayed to users by AIR.

REQ-041110: When annotations are deleted from previously annotated PDF standard conforming documents, AIR shall preserve all non-annotation components of the pre-existing PDF standard conforming documents whether or not the components are displayed to users by AIR.

5.    Other Nonfunctional Requirements

5.1    Performance Requirements

Note: There are no stated performance requirements for loading, saving, or displaying PDF content because those operations are primarily dependent on computer system performance characteristics outside the control of the AIR developers. For example, saving and loading may take place over slow network connections. Display may be remote using X windows over a slow remote network connection. Loading, saving, and displaying operations performed by AIR will be generally consistent with similar operations performed by similar applications running on the same computer system. Fore example, if Adobe Acrobat requires 1.0 seconds to load and display a PDF document, AIR will consume a similar amount of time.

REQ-051000: AIR shall consume no more than 0.1 seconds of user time to create any annotation in an existing PDF document. User time will be measured with the Unix time program on a computer system running Linux for x86 or x86-64 with at least one processor running at 2.0 GHz or faster. Ref. https://en.wikipedia.org/wiki/Time_(Unix)#User_Time_vs_System_Time

REQ-051010: AIR shall consume no more than 0.1 seconds of user time to delete any annotation in an existing PDF document. User time will be measured with the Unix time program on a computer system running Linux for x86 or x86-64 with at least one processor running at 2.0 GHz or faster. Ref. https://en.wikipedia.org/wiki/Time_(Unix)#User_Time_vs_System_Time

REQ-051020: AIR shall consume no more than 0.1 seconds of user time to resize any box annotation in an existing PDF document. User time will be measured with the Unix time program on a computer system running Linux for x86 or x86-64 with at least one processor running at 2.0 GHz or faster. Ref. https://en.wikipedia.org/wiki/Time_(Unix)#User_Time_vs_System_Time

Appendix A: Glossary

Annotation, annotating, annotate, annotated - refers to text and boxes added via AIR to a pre-existing PDF document.

Box, Boxes – refers to rectangular outlines or filled areas. Each rectangular area is no larger that a single page where page is defined by Portable Document Format First Edition July 1 2008 Section 7.7 Document Structure.

Creation, creating, create, created – refers to insertion/addition of annotations not previously in a PDF document. Creating annotations must preserve pre-existing content in the PDF document.

Deletion, deleting, delete, deleted – refers to removing annotations previously created in a PDF document via AIR. Deleting annotations must preserve pre-existing non-annotation content in the PDF document.

Loading, load, loaded - refers to display of non-annotation text as well as annotations within PDF documents selected from within a file system by a user. Loading must preserve pre-existing non-annotation content in the PDF document.

Modification, modifying, modify, modified – refers exclusively to changes in position, size, or character sequences in annotations.

Non-annotation content – refers to any PDF standard conforming information other than annotations that may be stored within a PDF document.

Preserving, preserve, preserved - refers to not semantically altering non-annotation content within a PDF file. Non-semantic changes like reordering the definitions of components within PDF files are permissible as long as the resulting PDF document remains PDF standard conferment. Ref. Section 2.2.5 of "PDF Reference sixth edition" version 1.7 November 2006 which states, "A PDF file should be thought of as a flattened representation of a data structure consisting of a collection of objects that can refer to each other in any arbitrary way. The order of the objects occurrence in the PDF file has no semantic significance."

Saving, save, saved– refers to storing PDF documents with user selected file names at user selected file system locations.

Text - refers to "Text Objects" described by Section 5.3 of "PDF Reference sixth edition" version 1.7 November 2006 with the added restriction that text" is composed solely of sequences of Latin characters irrespective of font specified in the PDF for display of the characters.

User Time – refers to the 'user CPU time'. Total CPU time is the combination of the amount of time the CPU(s) spent performing some action for a program and the amount of time the CPU(s) spent performing system calls for the kernel on the program's behalf. User CPU time is used as opposed to wall clock time, the actual elapsed time measured with a stopwatch, because modern time-share computer systems allocate unpredictable percentages of processor execution time to running processes like AIR. Ref: https://en.wikipedia.org/wiki/Time_(Unix)#User_Time_vs_System_Time. Measured Wall clock time will be repeatable for interactive programs like AIR, but an upper limit to User Time can be verified and is repeatable.
